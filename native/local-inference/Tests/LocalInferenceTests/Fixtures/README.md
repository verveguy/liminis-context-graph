# Integration Test Fixtures

This directory contains the fixture assets loaded by `LocalInferenceIntegrationTests.swift`
and `EmbeddingOutputValidationTests.swift`.

## Fixture inventory

Every asset in this directory is either **production-derived** (generated by the same
script that prepares the production build) or a **documented stub** (a deliberately
minimal asset for a narrow test boundary, with a written rationale below).

| Asset | Type | Generator | Notes |
|-------|------|-----------|-------|
| `tokenizer-cache/` | Production-derived | `prepare-tokenizer.py` | Staged tokenizer files for offline `swift-transformers` loading. Pinned to `PINNED_BGE_REVISION`. |
| `stub-bge-base.mlpackage` | Documented stub | `generate-stub-model.py --precision fp32` | Tiny synthetic CoreML model. See "Stub rationale" below. |
| `stub-bge-base-fp16.mlpackage` | Documented stub | `generate-stub-model.py --precision fp16` | fp16 variant. Same rationale. |
| `stub-bge-base-bad-dtype.mlpackage` | Documented stub | `generate-bad-stub-models.py` | Exercises `embeddingOutputDtypeUnsupported` in `validateOutputSchema`. |
| `stub-bge-base-bad-shape.mlpackage` | Documented stub | `generate-bad-stub-models.py` | Exercises `embeddingOutputShapeMismatch` in `validateOutputSchema`. |
| `stub-bge-base-bad-output-name.mlpackage` | Documented stub | `generate-bad-stub-models.py` | Exercises `embeddingOutputMissing` in `validateOutputSchema`. |
| `generate-stub-model.py` | Generator script | — | Source of the two positive-path stubs. |
| `generate-bad-stub-models.py` | Generator script | — | Source of the three negative-path stubs. |
| `requirements.txt` | Pinned deps | — | Exact `coremltools` and `numpy` versions for byte-identical stub generation. |
| `*.script-hash` | Sentinel files | `check-fixture-freshness.sh` / `refresh-test-fixtures.sh` | See "Divergence guard" below. |

## Stub rationale

`stub-bge-base.mlpackage` and `stub-bge-base-fp16.mlpackage` are synthetic CoreML models
generated by `generate-stub-model.py`. They are **not** produced by `convert-embedding-model.py`
(the production conversion script).

**Why stubs instead of the real model?**
- `convert-embedding-model.py` produces an 85–90 MB model, requires HuggingFace network
  access, and takes 2–5 minutes per run. Committing it would bloat the repo and tie CI
  to upstream availability.
- Both historical bugs (783, 784) were I/O *contract* bugs — wrong dtype, missing
  tokenizer file — not numerical correctness bugs. The stubs faithfully mirror the
  production I/O contract (same input/output names, shapes, and precision options), which
  is exactly what the integration tests validate.
- The script-hash guard on `convert-embedding-model.py` closes the gap: if the production
  script changes its output interface, `check-fixture-freshness.sh` fires, prompting the
  developer to check whether `generate-stub-model.py` also needs updating.

The three negative-path stubs (`bad-dtype`, `bad-shape`, `bad-output-name`) are
*deliberately wrong* — they cannot be produced by any valid production conversion. They
exist solely to exercise the three schema-validation error paths in
`CoreMLEmbeddingActor.validateOutputSchema`. Hand-rolling them is appropriate and intended.

## Divergence guard

Four sentinel files (`.script-hash`) track whether the committed fixtures match the current
versions of their generator scripts:

| Sentinel | Tracks |
|----------|--------|
| `tokenizer-cache/.script-hash` | `prepare-tokenizer.py` |
| `generate-stub-model.script-hash` | `generate-stub-model.py` |
| `generate-bad-stub-models.script-hash` | `generate-bad-stub-models.py` |
| `convert-embedding-model.script-hash` | `convert-embedding-model.py` |

Each sentinel contains the SHA-256 hash of its tracked script. `check-fixture-freshness.sh`
compares the current script hash against the committed sentinel — if they differ, it exits
non-zero with a targeted error message. This check runs as a CI job (`fixture-freshness-check`
in `.github/workflows/test.yml`) on every PR.

## Refreshing fixtures

After editing any generator script, run:

```bash
# From liminis-app/native/local-inference/
bash refresh-test-fixtures.sh
```

Or via pnpm from `liminis-app/`:

```bash
pnpm refresh:test-fixtures
```

The script regenerates all fixtures and rewrites all four sentinels. It requires:
- `uv` on PATH
- Network access to HuggingFace (for the tokenizer step; revision is pinned)
- macOS (CoreML is macOS-only; the mlpackage stub steps won't run on Linux)

After running, review the diff and commit the updated fixtures alongside the script change.

## Checking freshness locally

```bash
# From liminis-app/native/local-inference/
bash check-fixture-freshness.sh

# Or via pnpm from liminis-app/
pnpm check:fixture-freshness
```

## Bug retrospective (FR-009)

This fixture discipline was introduced after two production-blocking bugs slipped through
CI because hand-rolled fixtures masked them.

### Bug 783 — tokenizer offline cache (missing `config.json`)

**What happened**: The initial tokenizer staging omitted `config.json` from the staged
file set. The hand-rolled fixture had `config.json` present, so integration tests passed.
The production sidecar failed at startup with `offlineModeError` because
`swift-transformers` requires `config.json` for `AutoTokenizer.from()`.

**How the new system catches it**: `prepare-tokenizer.py` owns the list of required files
(`TOKENIZER_FILES`). If a PR drops `config.json` from that list, `prepare-tokenizer.py`
changes. The CI `fixture-freshness-check` job compares the current `prepare-tokenizer.py`
hash against `tokenizer-cache/.script-hash` — they diverge, CI fails, and the error
message names the changed script.

### Bug 784 — embedding output dtype (fp32 vs fp16)

**What happened**: The production `convert-embedding-model.py` was updated to emit fp16
output (matching the CoreML best practice for on-device inference). The legacy stub was
fp32. `CoreMLEmbeddingActor` received fp16 tensors but tried to read them as fp32, producing
zero or NaN embeddings silently. Integration tests passed because the fp32 stub still matched
the old actor code. The mismatch was only found during end-to-end testing.

**How the new system catches it**: `convert-embedding-model.py` is now tracked by
`convert-embedding-model.script-hash`. If a PR changes the production conversion (e.g.,
switches precision or renames the output feature), that script's hash changes, CI fires,
and the developer is told to run `refresh-test-fixtures.sh` — which includes updating
`generate-stub-model.py` to match the new I/O contract. Because `generate-stub-model.py`
is also sentinel-tracked, a developer who updates the production script but not the stub
generator gets a second CI failure pointing at the stub generator.
